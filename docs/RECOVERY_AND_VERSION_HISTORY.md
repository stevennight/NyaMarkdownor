# 工作恢复与版本历史设计规格

[中文首页](../README.zh-CN.md) | [架构说明](ARCHITECTURE.zh-CN.md) | [Architecture](ARCHITECTURE.md)

## 文档状态

本文适用于 NyaMarkdownor `1.0.3`，定义并记录当前实现遵循的工作恢复、版本历史、外部文件冲突和保留策略。

本文中的“版本历史”只有一条用户可见的时间线。`automatic`、`safety` 和 `manual` 是同一条时间线中的保留类别，不是三个互相独立的功能入口。

用户界面统一展示历史，但物理存储分为 Rust 管理的磁盘版本和前端管理、镜像到原生 app-data 的本地编辑器检查点。本文会明确区分概念类别与实际记录字段。

## 设计目标

- 正常退出或意外退出后，可以继续最近的编辑现场。
- 普通保存频繁发生时，不产生大量永久历史版本。
- 关闭、重新加载、恢复和覆盖等可能丢失内容的动作发生前，保留固定版本。
- 自动版本、重大变动版本和用户手动版本采用独立的保留与淘汰规则。
- 检测到外部文件修改时，不允许自动保存静默覆盖磁盘内容。
- 历史容量不足时，在写入文件前让用户明确选择，不静默牺牲用户手动版本。

## 概念边界

### 工作恢复

工作恢复保存“最近的工作现场”，用于重启后继续编辑。它包括打开的标签、活动标签、编辑器内容、最后确认的磁盘基线、光标和可恢复的编辑器状态。

工作恢复不是版本历史：

- 每个标签只维护最新状态，后一次状态覆盖前一次状态。
- 普通输入、粘贴和查找替换只更新工作恢复，不创建历史版本。
- 正常启动并恢复工作现场时，不额外创建历史版本。
- 正常关闭整个应用时只强制刷新工作恢复，不创建“关闭前版本”。

### 版本历史

版本历史保存可供用户比较和恢复的内容。所有历史条目展示在同一条时间线中，并通过类别、状态和原因解释来源。

| 类别 | 状态 | 用途 |
| --- | --- | --- |
| `automatic` | `rolling` 或 `fixed` | 自动保存和 `Ctrl+S`/手动保存产生的普通保存版本 |
| `safety` | `fixed` | 重新加载、恢复、关闭脏标签、覆盖目标或覆盖外部修改前的重大变动版本 |
| `manual` | `fixed` | 用户主动创建并希望长期保留的重要节点 |

只有 `automatic` 可以处于 `rolling` 状态。`safety` 和 `manual` 创建后立即固定。

### 版本历史界面

- 版本历史侧边栏只显示当前文档的磁盘版本和本地检查点，不混入其他文档或孤立文件的历史。
- 侧边栏提供“管理文件历史”入口，打开与设置窗口同尺寸、同导航结构的独立管理框。
- 管理框一级分为“文档历史”和“孤立文件历史”。孤立文件仅指已经明确确认源文件缺失的历史；无路径草稿或暂时无法确认状态的文档不归为孤立。
- 两个分类都先按文档列出汇总信息，再进入完全相同的版本详情列表。详情按时间统一排列磁盘版本和本地检查点。
- 文档列表支持删除该文档的全部磁盘版本和本地检查点；版本详情支持单独删除一个版本。删除历史不修改源文件或当前编辑器内容。
- 任意磁盘版本或本地检查点都可作为独立草稿打开，避免管理其他文档时误覆盖当前编辑内容。
- 新版中央历史具有全局索引。旧版相邻 `.nyamarkdownor-backups` 没有全局索引，仅在源路径仍可由当前标签、最近文件或本地检查点得知时并入管理；一旦并入，汇总、单版本删除和整文档删除与新版历史一致。

### 写入安全

临时文件、原子替换和写入失败回滚属于文件写入安全机制，不显示在版本历史中，也不计入本文的检查点数量。

## 工作恢复数据模型

有文件路径的标签必须同时保存以下三值模型中的前两项；第三项在恢复或写入前从磁盘读取：

| 值 | 持久化方式 | 含义 |
| --- | --- | --- |
| `baseContent` | 写入工作恢复文件 | 最近一次由应用确认的磁盘内容 |
| `editorContent` | 写入工作恢复文件 | 编辑器当前内容，可能尚未保存 |
| `diskContent` | 启动或保存前读取 | 当前磁盘上的实际内容 |

工作恢复记录还必须包含：

- 稳定的标签或文档 ID。
- 规范化文件路径；未命名草稿为 `null`。
- `baseContent` 的内容哈希。
- 最近确认的文件修改时间和大小。
- 工作恢复记录的更新时间。
- 光标、选区和可恢复的标签级编辑状态。

文件时间和大小用于快速发现变化，内容比较或内容哈希才是最终判断依据。时间或大小相同不能单独证明文件未变。

### 启动恢复决策

启动时必须先完成恢复与磁盘冲突判断，再启用自动保存。

| 条件 | 行为 |
| --- | --- |
| 未命名草稿 | 恢复 `editorContent`，保持未保存状态 |
| `diskContent == baseContent` | 恢复 `editorContent`；`editorContent != diskContent` 时标记未保存 |
| `diskContent != baseContent` 且 `editorContent == diskContent` | 内容没有实际分叉，可直接使用当前磁盘内容并标记为已保存 |
| `diskContent != baseContent` 且 `editorContent != diskContent` | 暂停自动保存，提供比较，并让用户选择使用磁盘内容或恢复编辑内容 |
| 文件缺失或暂时无法读取 | 不把读取失败当作空文件；恢复编辑内容，标记磁盘不可用，并暂停对原路径的自动保存，等待用户重新定位、另存为或重试 |

发生实际分叉时：

- 选择“使用磁盘内容”：若恢复记录中的编辑内容与当前磁盘内容不同，先将被放弃的 `editorContent` 写入固定 `safety` 检查点，原因记为 `recovery-discard`；随后在编辑器中加载 `diskContent`，状态为已保存。
- 选择“恢复编辑内容”：在编辑器中加载 `editorContent`，不立即写入磁盘；是否未保存以 `editorContent != diskContent` 为准。
- 选择“比较”：打开两份内容的差异视图，不解除冲突，也不启动自动保存。

工作恢复本身不创建历史版本。只有用户选择磁盘内容并实际放弃一份不同的恢复内容时，才创建上述固定安全版本。

## 历史检查点数据模型

版本历史界面把两种物理记录合并为按文档排列的时间线，但两者不能混为同一种磁盘格式。

### 磁盘版本

磁盘版本由 Rust 中央历史管理：

- 以规范化源路径生成来源桶；桶内 `source.json` 记录源路径、文件名、最后确认时间和孤立时间。
- 版本文件名编码来源前缀、`rolling | automatic | safety | manual` 类别和纳秒时间。
- `rolling` 同时保存滚动窗口开始时间和最后更新时间；其他类别创建后固定。
- 返回前端的记录包含路径、名称、开始时间、更新时间、大小和类别。
- 磁盘版本不持久化编辑器原因数组；`safety` 的具体来源由触发保存流程决定，界面显示为安全检查点。

### 本地编辑器检查点

本地检查点的实际 `DraftSnapshot` 记录包含：

- `id`
- `documentId`；有文件路径的文档为 `null`，无路径草稿使用稳定标签 ID
- `fileName`、`filePath`
- `markdown`、`lastSavedMarkdown`、`lineEnding`、`fileStats`
- `createdAt`、`size`、`contentHash`
- `kind`：`safety | manual`
- `reason` 和带发生时间的 `reasons`

记录容器还包含格式版本、`tableCellBreakFormat: "html"` 和 `savedAt`，用于确保表格单元格换行以 `<br>` 语义恢复。

### 统一概念类别

界面和保留策略把 `rolling` 视为尚未固定的 `automatic`。固定类别的保留优先级为：

类别的保留优先级为：

```text
manual > safety > automatic
```

同一来源的精确相同内容不创建重复可见记录。磁盘版本通过字节比较复用或提升已有固定版本；本地检查点通过 `contentHash` 和完整 Markdown 比较合并原因。后续事件只允许提升保留级别，不允许降低已有类别。

### 原因值

本地编辑器检查点使用以下稳定原因值，界面显示本地化名称：

| 类别 | 原因 | 含义 |
| --- | --- | --- |
| `manual` | `manual` | 用户主动创建检查点时的编辑器内容 |
| `safety` | `recovery-discard` | 选择磁盘内容前被放弃的恢复内容 |
| `safety` | `reload` | 从磁盘重新加载前的未保存编辑内容 |
| `safety` | `restore` | 恢复历史版本或检查点前的未保存编辑内容 |
| `safety` | `close` | 关闭脏标签前的编辑内容 |
| `safety` | `save-conflict` | 处理保存冲突前保留的编辑内容 |
| `safety` | `save-as-overwrite` | 可识别的另存为覆盖前编辑内容原因；当前覆盖目标主要由磁盘 `safety` 版本保护 |

`save-as-overwrite`、`window-close`、`legacy-idle` 和 `legacy-preserved` 保留为可识别记录值，但当前常规流程不会创建这些本地原因。正常关闭窗口只刷新工作恢复，不创建窗口关闭检查点；覆盖已有 Save As 目标时，目标原磁盘内容写为磁盘 `safety` 版本。普通保存产生的磁盘 `rolling/automatic` 版本没有本地 `reason` 字段。

## 普通保存与滚动版本

自动保存和 `Ctrl+S`/手动保存使用同一套版本规则。两者只在原因标签上不同，不使用不同的保留队列。

默认检查点间隔为 10 分钟，并允许在设置中调整。

每个文档最多维护一个明确的“进行中的自动检查点”指针。不能通过“时间线最新条目是否为手动版本”推断滚动状态。

### 保存算法

当编辑器内容与当前磁盘内容相同，不写文件、不创建检查点，也不更新滚动窗口。

需要写入时，依次执行：

1. 读取并核对当前磁盘内容，先处理外部修改冲突。
2. 将写入前的当前磁盘内容作为本次 `automatic` 候选内容。
3. 若不存在进行中的自动检查点，创建 `rolling` 检查点，并同时设置 `startedAt` 和 `updatedAt` 为当前时间。
4. 若存在进行中的自动检查点，且 `now - startedAt < 检查点间隔`，用本次候选内容更新该检查点，并只更新 `updatedAt`。
5. 若存在进行中的自动检查点，且 `now - startedAt >= 检查点间隔`，先将原检查点固定，再用本次候选内容创建新的 `rolling` 检查点。
6. 历史保护成功，或用户明确选择“本次不创建历史仍继续保存”后，才写入目标文件。
7. 文件写入成功后，更新工作恢复中的 `baseContent`、编辑器保存基线、文件时间和大小。

更新滚动内容绝不能重置 `startedAt`，否则持续保存会导致滚动检查点永远无法固定。`updatedAt` 表示滚动内容最后一次更新的时间。

### 固定事件与滚动指针

创建任何 `safety` 或 `manual` 检查点前：

1. 若存在进行中的 `automatic` 检查点，立即将它固定，即使尚未达到检查点间隔。
2. 清除进行中的自动检查点指针。
3. 创建或复用本次固定检查点。

下一次普通保存再开始新的滚动检查点。这样手动检查点和重大变动检查点不会被后续普通保存覆盖。

## 完整触发规则

| 操作 | 最终规则 |
| --- | --- |
| 输入、粘贴、查找替换 | 更新工作恢复中的编辑器内容；500 毫秒空闲防抖，持续编辑最长每 5 秒落盘；不创建历史版本 |
| 正常退出应用 | 强制刷新完整工作恢复记录；不创建历史版本；重启后继续恢复打开的标签和未保存内容 |
| 意外退出应用 | 使用最近一次周期性工作恢复记录；不因重启本身创建历史版本 |
| 工作恢复 | 以恢复记录中的磁盘基线、编辑器内容和当前磁盘内容做三值核对；有分叉时暂停自动保存并允许比较。选择磁盘内容时，先把被放弃的恢复内容写为固定重大变动检查点；选择恢复内容时载入编辑器并按它与当前磁盘内容的差异标记未保存 |
| 自动保存 | 仅在内容变化且没有未解决外部冲突时执行；保存前磁盘内容写入或更新进行中的滚动自动版本。外部修改或历史空间不足时暂停，不静默覆盖、不自行跳过历史 |
| `Ctrl+S`/手动保存 | 内容无变化时无操作；有变化时与自动保存使用同一滚动规则。历史空间不足时让用户选择打开版本历史、本次无历史保存或取消 |
| 手动创建检查点 | 先固定进行中的自动版本；立即以当前编辑器内容创建固定手动检查点；不受检查点间隔限制；达到手动数量或全局硬限制时打开历史清理或取消创建 |
| 重新加载磁盘文件 | 若有将被替换的未保存内容，先固定自动版本并创建固定重大变动检查点；成功后才加载磁盘内容并标记已保存。检查点失败则取消并提供历史管理入口 |
| 恢复历史版本 | 若有将被替换的未保存内容，先创建固定重大变动检查点；随后只把所选内容载入编辑器，不直接写磁盘；检查点失败则取消并提供历史管理入口 |
| 关闭脏标签 | 先固定自动版本并立即创建固定 `关闭前` 重大变动检查点；成功后才从工作恢复中移除标签；不走滚动间隔。检查点失败则取消关闭并提供历史管理入口 |
| 关闭干净标签 | 从工作恢复会话中移除标签；不创建历史版本 |
| 另存为到新目标 | 写入新文件并把编辑器绑定到新目标；目标没有旧内容时不创建覆盖前检查点 |
| 另存为覆盖已有目标 | 先固定自动版本；把目标文件原内容写为目标文件的固定重大变动检查点；写入成功后编辑器绑定目标文件，源历史不复制。容量不足时提供打开历史、本次无历史保存或取消 |
| 自动保存发现外部修改 | 立即暂停自动保存，不覆盖磁盘，提供比较、重新加载或用户明确覆盖的流程 |
| 用户确认覆盖外部修改 | 先固定自动版本；把当前外部磁盘内容写为固定重大变动检查点；成功后才用编辑器内容覆盖磁盘 |
| 内容未变化时保存 | 不写文件，不创建或更新时间线条目 |

批量保存必须对每个文档独立执行同一规则，不能跳过单个文件的外部修改检查和容量决策。

### 工作恢复写入节奏

工作恢复写入默认采用 500 毫秒空闲防抖，并保证连续编辑最长每 5 秒至少持久化一次。正常退出时必须强制刷新，不等待防抖计时器。意外退出最多丢失最近一次成功持久化之后的编辑内容。

## 检查点内容来源

| 场景 | 写入历史的内容 | 归属 |
| --- | --- | --- |
| 自动保存 | 写入前的当前磁盘内容 | 当前文件的 `automatic` |
| `Ctrl+S`/手动保存 | 写入前的当前磁盘内容 | 当前文件的 `automatic` |
| 手动创建检查点 | 当前编辑器内容 | 当前文档的 `manual` |
| 启动时选择磁盘内容 | 被放弃的恢复编辑内容 | 当前文档的 `safety` |
| 重新加载磁盘文件 | 被替换的未保存编辑内容 | 当前文件的 `safety` |
| 恢复历史版本 | 被替换的未保存编辑内容 | 当前文件的 `safety` |
| 关闭脏标签 | 关闭前的编辑器内容 | 被关闭文档的 `safety` |
| 另存为覆盖已有目标 | 目标文件被覆盖前的磁盘内容 | 目标文件的 `safety` |
| 覆盖外部磁盘修改 | 外部修改后的当前磁盘内容 | 当前文件的 `safety` |

恢复某个历史版本不会自动把该历史版本再复制一份。只有恢复动作会覆盖一份不同的未保存编辑内容时，才保护被覆盖的编辑内容。

## 另存为规则

- “另存为”写入成功后，编辑器绑定到目标文件，后续保存和历史都归属目标文件。
- 覆盖已有目标时，目标原内容必须先保存为目标文件的固定安全检查点。
- 源文件保留自己的历史，不把整套源文件历史复制到目标文件。
- 另存为失败时，编辑器仍绑定原文件，不能提前改变保存基线或历史归属。
- 若产品同时提供“保存副本”，保存副本不改变当前编辑器绑定；但覆盖已有副本目标时仍需保护目标原内容。

## 外部文件修改规则

每次写入已有路径前都必须将当前磁盘内容与应用最后确认的 `baseContent` 比较。修改时间和大小只用于快速检测，最终以内容或可靠内容哈希判断。

发现外部修改后：

- 自动保存立即暂停，不能静默覆盖。
- 用户可以比较编辑器内容与磁盘内容。
- 用户选择重新加载磁盘时，按“重新加载磁盘文件”规则保护未保存编辑内容。
- 用户选择覆盖磁盘时，先把当前外部磁盘内容保存为固定的磁盘 `safety` 版本，再执行写入。
- 用户可以改用另存为，原路径及其外部内容保持不变。
- 冲突未解决前，工作恢复继续更新编辑器内容，但不能把冲突误标记为已保存。

## 内容去重

所有类别都按同一来源下的精确内容哈希去重。哈希基于实际可恢复的内容字节计算。

当候选内容已经存在时：

- 不创建第二份内容相同的可见历史条目。
- 保留原始 `createdAt` 和原有原因，并追加新的带时间原因。
- 更新 `updatedAt` 或最近使用时间。
- 若新原因要求更高保留级别，按 `automatic -> safety -> manual` 提升类别。
- 若现有记录处于 `rolling`，而新事件要求固定版本，则立即固定并清除滚动指针。
- 一条记录只计入其当前最高保留类别的队列，全局数量也只计算一次。

不同来源文件之间默认不合并可见检查点；实现可以在存储层按内容哈希共享不可变数据，但不能混淆文件归属和恢复时间线。

## 保留与淘汰

三类检查点分别计算、分别淘汰。以下值为默认设置，均允许用户调整：

| 类别 | 每文件最大数量 | 全局保留队列 | 默认保留天数 | 自动淘汰 |
| --- | ---: | --- | ---: | --- |
| `automatic` | 48，包含至多一个滚动版本 | 独立统计 | 180 天 | 可以；先删除过期或最旧的固定自动版本 |
| `safety` | 32 | 独立统计 | 365 天 | 可以；先删除过期版本，超过数量限制时再删除最旧版本 |
| `manual` | 32 | 独立统计 | 0，表示永不过期 | 不允许因数量或容量限制静默淘汰；达到数量限制时让用户选择删除现有手动版本或取消创建。用户显式设为非零保留天数时，过期版本可按该设置清理 |

共享的默认全局硬限制为：

- 全部类别合计最多 2,048 个检查点。
- 全部历史内容合计最多 2 GiB。
- 单个候选检查点最多 256 MiB。

全局数量和容量是共享硬限制，但清理候选仍按类别队列选择，不能用普通自动保存无差别挤占手动版本。

### 容量提示

当检查点数量或历史容量达到对应硬限制的 80% 时显示非阻塞提示，并指向现有的版本历史入口；提示本身不触发立即清理。

### 自动清理顺序

创建新检查点前，按以下顺序清理允许自动淘汰的内容：

1. 已过期的 `automatic` 版本。
2. 超过每文件数量限制或全局压力下最旧的固定 `automatic` 版本；必要时可放弃最旧的滚动自动版本。
3. 已过期的 `safety` 版本。
4. 超过每文件数量限制的最旧 `safety` 版本。

不得静默删除 `manual` 版本。未过期且未超过自身数量限制的 `safety` 版本也不为普通保存让路；历史管理界面中的用户明确删除不受此限制。

### 达到硬限制

完成允许的自动清理后仍无法为检查点腾出空间，或单个候选内容超过限制时，必须在写入当前文件前暂停操作并提示用户选择：

1. 打开历史管理并清理版本，然后重试。
2. 对保存操作，明确确认“本次不创建历史，仍继续保存”。该授权只对当前一次操作有效。关闭脏标签、重新加载和恢复历史等丢弃/替换动作没有此选项，必须先成功创建检查点。
3. 取消保存，保持编辑器未保存状态。

自动保存遇到该情况时同样暂停，不能自行选择“无历史保存”。文件写入和历史写入必须保持明确的先后关系，不能在历史保护失败后静默覆盖文件。

## 界面呈现

用户只看到一条“版本历史”时间线：

- `automatic + rolling`：显示为“自动版本（更新中）”。
- `automatic + fixed`：显示为“自动版本”。
- `safety + fixed`：根据原因显示“关闭前”“恢复前”“覆盖外部修改前”等明确标签。
- `manual + fixed`：显示为“手动检查点”，并允许用户删除。

工作恢复不作为另一组历史列表显示。只有被放弃、替换或关闭的内容真正转成固定安全检查点后，才进入版本历史。

## 实现验收条件

- 连续普通保存只更新一个滚动自动版本，`startedAt` 不随更新改变。
- 到达检查点间隔后的下一次普通保存会固定旧滚动版本并开始新滚动版本。
- 手动或重大变动检查点不会被后续普通保存覆盖。
- 内容未变化时反复按 `Ctrl+S` 不增加历史。
- 关闭脏标签一定产生固定安全检查点；正常关闭整个应用不产生历史版本。
- 重启恢复本身不产生历史；选择磁盘并放弃不同恢复内容时才产生固定安全检查点。
- 自动保存检测到外部修改时停止，直到用户作出选择。
- 另存为覆盖目标时，目标原内容进入目标历史，源历史不复制。
- 相同内容不会产生重复条目，较高保留类别可以提升已有条目。
- 自动和安全版本按各自限制清理，手动版本不会被静默删除。
- 达到硬限制时，文件不会在用户选择之前被覆盖。
