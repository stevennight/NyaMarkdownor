# NyaMarkdownor 架构说明

[English](ARCHITECTURE.md) | [简体中文](ARCHITECTURE.zh-CN.md)

本文说明 NyaMarkdownor `1.0.3` 已实现的架构、数据归属和必须保持的产品边界。后续方向会明确标注；其余内容均描述当前行为。

## 产品目标

NyaMarkdownor 是本地优先的跨平台 Markdown 桌面编辑器。产品目标是在保持日常写作流畅度的同时，加强表格编辑、选择与复制、本地文件安全和长文档性能。

界面需要像真正的桌面生产工具：信息密度足够、适合长时间写作，并且编辑反馈不能脱离用户输入。

## 技术栈

- 桌面外壳：Tauri 2。
- 原生层：Rust 文件 IO、工作区扫描、版本历史和系统集成。
- 前端：React、TypeScript、Vite。
- 规范源码编辑器：CodeMirror 6。
- 所见即所得投影：Tiptap。
- Markdown 预览：markdown-it，默认禁用原始 HTML。
- 图标与界面：lucide-react 和项目自有 CSS。

Tauri 让应用保持较小的安装体积，同时提供原生文件能力；Rust 负责不能交给 WebView 的文件安全边界；CodeMirror 负责增量、大文本源码编辑；Tiptap 只作为 Markdown 的结构化投影，不成为新的持久化格式。

## 核心数据原则

- Markdown 源文本是唯一规范文档表示。
- 源码、预览和所见即所得模式都必须围绕同一份 Markdown 工作。
- 不把 Tiptap、CodeMirror 或其他编辑器实现细节写入 Markdown、恢复记录、剪贴板或磁盘格式。
- 所见即所得模式支持的结构必须可回写为等价 Markdown；暂不支持安全编辑的结构使用受保护节点显示，不能静默删除或改写。
- 表格单元格换行使用 `<br>`；物理换行始终结束 Markdown 表格行。
- 原始 HTML 默认禁用，表格 `<br>` 只走受限的安全渲染路径。

## 运行时边界

### 桌面版

桌面版是完整产品运行时：

- 原生打开、创建、保存和导出。
- 最近文件、文件夹工作区和启动/打开方式文件。
- 本地图片相对路径插入。
- 磁盘外部修改检查。
- 工作恢复、版本历史和备份容量管理。
- 系统文件关联与文件管理器操作。

文件 IO、路径验证、编码解码、原子替换、备份和工作区扫描由 Rust 完成。WebView 不直接拥有任意本地文件权限。

### 网页预览

网页预览仅用于界面和前端开发：

- 新建产生未绑定草稿。
- 打开显示为“导入草稿”。
- 保存和另存为只下载副本。
- 不提供真实磁盘绑定、工作区、最近文件、版本历史或外部修改安全保证。

这种限制是刻意设计，防止网页环境伪装成已绑定本地路径的桌面编辑器。

## 文档与标签页模型

每个文档标签拥有独立的：

- Markdown 文本和最后保存基线。
- 文件路径或 `null` 草稿身份。
- 原始换行风格。
- 最近确认的文件时间/大小。
- CodeMirror 选择、滚动和撤销/重做状态。
- 所见即所得模式的标签级历史快照。

打开文件、最近文件、工作区文件和草稿都进入统一标签页。打开真实文件时可复用干净的空白占位标签；同一路径不会重复打开。多选打开采用尽力而为策略，一个文件失败不会取消其他可读文件。

React 活动标签状态与当前挂载编辑器所有者需要分别追踪，避免快速切换时把旧编辑器的最后文本写入新标签。

关闭脏标签必须先成功创建固定安全检查点；关闭整个窗口只刷新完整工作恢复记录，不额外创建历史版本。

## 编辑投影

### 源码模式

CodeMirror 直接编辑规范 Markdown。格式化、查找替换、表格结构操作和预览交互最终都必须生成局部 Markdown 文本变更。

### 预览模式

markdown-it 在 Web Worker 中渲染经过清理的 HTML。相对图片由桌面层解析为本地资源 URL；相对 Markdown 链接通过应用的本地文件流程打开，而不是让 WebView 离开编辑器页面。

大文档会暂停自动完整预览并进入显式快照模式，标题提取继续以更低频率运行。

### 所见即所得模式

Tiptap 从 Markdown 解析结构，并在离开/保存时序列化回 Markdown。前言、标题、强调、代码、围栏、列表、任务列表、链接、图片、表格等支持结构需要接受往返测试。缓存原始写法只能用于保真，语义变化后必须失效。

## 表格与剪贴板

表格编辑基于共享的 Markdown 表格解析和单元格边界逻辑：

- 表格检测跳过围栏代码和缩进代码。
- 行列增删、复制、移动、排序、对齐和粘贴只替换目标表格范围。
- Table Inspector 的焦点与 CodeMirror 结构化选择双向同步。
- TSV、CSV、HTML、`text/markdown` 和普通 Markdown 表格使用同一粘贴识别路径。
- 结构化选择可表示单元格、矩形区域、整行、整列、表头、正文和整表。
- 删除或剪切单元格只清空内容，不破坏 Markdown 竖线结构。

智能复制根据选择上下文同时产生：

- `text/plain`：干净文本或网格友好内容。
- `text/html`：经过清理的富文本或真实表格。
- `text/markdown`：规范 Markdown 片段。

普通单词级选择仍按源码文本处理，不能因为位于表格中就被错误扩展为整个单元格。

## 应用身份与数据目录

- 正式版 Bundle Identifier：`io.github.stevennight.nyamarkdownor`。
- 开发版 Bundle Identifier：`io.github.stevennight.nyamarkdownor.dev`。
- Tauri 根据标识符生成 app-data、app-local-data、缓存和 WebView 数据目录，因此正式版与开发版天然隔离。
- 当前产品按新的应用身份处理，不扫描或迁移旧 Bundle Identifier 的数据目录。

app-data 中保存带时间戳的 JSON 状态：

- 偏好设置。
- 完整文档标签会话。
- 活动草稿。
- 本地安全/手动检查点。
- 最近文件。
- 工作区根目录。

app-local-data 中的 `backups-v1` 保存中央磁盘版本历史。用户可在设置中选择其他备份根目录；切换后显式记录旧的自定义根目录，以便历史仍可访问。

## 文件安全与版本历史

### 写入顺序

1. 核对目标当前状态和预期文件统计。
2. 必要时把磁盘内容写入中央版本历史。
3. 通过临时文件写入并同步。
4. 原子发布新文件；失败时尝试回滚旧文件。
5. 写入成功后更新编辑器保存基线和工作恢复记录。

外部内容发生变化或路径无法确认时，自动保存暂停。用户必须比较、重新加载、确认覆盖或另存为，应用不能把读取失败当作空文件重建。

### 工作恢复

工作恢复保存最新工作现场，不是一条历史时间线。它覆盖式更新标签、活动文档、编辑内容和磁盘基线；启动时先完成恢复与磁盘冲突判断，再允许自动保存。

### 版本历史

- `rolling`：当前进行中的普通保存版本。
- `automatic`：固定后的普通保存版本。
- `safety`：重大变动前保护的固定版本。
- `manual`：用户主动创建的固定版本。

磁盘版本和本地检查点在界面中按文档合并展示，但物理存储不同：磁盘版本由 Rust 管理，本地编辑器检查点由前端序列化并镜像到原生 app-data。

默认策略为：

- 每个来源 48 个自动版本。
- 每个来源 32 个安全版本。
- 每个来源 32 个手动版本。
- 全局最多 2,048 个条目或 2 GiB。
- 单个候选版本最多 256 MiB。
- 自动版本保留 180 天，安全版本保留 365 天，手动版本默认不按天过期。
- 孤立历史宽限期 365 天。

手动版本不会被静默淘汰。安全版本只按自身过期/数量规则清理，不能为了普通保存任意牺牲未过期安全版本。

## 性能边界

- Markdown 预览和标题提取运行在 Web Worker。
- 隐藏预览不执行完整渲染。
- 分屏滚动同步使用 ref 和 `requestAnimationFrame`，不在每帧触发 React 渲染。
- 状态栏指标采用单次记忆化扫描；查找面板关闭时不扫描匹配项。
- 表格操作尽量围绕光标局部解析，并向 CodeMirror 提交局部变更。
- 工作区扫描有最大文件数，前端列表也有可见数量上限。
- 命令面板只在打开时构建完整命令/文件候选。

## 安全边界

- 本地文件不上传。
- 不要求账号，不包含同步或遥测。
- 原始 HTML 默认关闭。
- 外部链接只有明确的 Ctrl/Cmd 点击才交给系统打开。
- 本地资源路径由桌面层转换，远程/data/blob 等 URL 不做本地重写。
- 删除历史和覆盖内容使用应用内确认界面。
- 历史桶删除前校验目录、元数据、未知文件、子目录、链接和重解析点，避免越界删除。

## 当前状态与后续方向

`1.0.3` 已包含桌面外壳、源码/预览/所见即所得编辑、结构化表格、智能剪贴板、多标签恢复、外部修改检查、统一版本历史、可配置保留策略以及英文/简体中文界面。

后续架构改进应遵循以下方向：

1. 只有在 Markdown 语义可保留时扩展所见即所得结构支持。
2. 依据性能测量把更多昂贵分析或导出工作迁移到 Worker/Rust。
3. 在保留显式快照降级路径的前提下改进超大文档增量预览。
4. 若引入版本父子关系，必须使用新的显式历史格式，不能把编辑器实现元数据塞进现有时间线。
